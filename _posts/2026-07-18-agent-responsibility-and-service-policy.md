**如果 Agent 开始持续承担责任，产品的基本单位也会变。**

传统软件围绕功能：发邮件、查航班、建日程、改代码。聊天产品围绕消息：User Message → Assistant Message。

Agent 产品更自然的单位，可能是：

**持续目标、任务或责任域。**

例如：管理我的出差，保证项目按期推进，控制云资源成本，维护仓库健康，处理低风险工单。

用户不再不断找功能入口，而是把一类责任交给系统。产品真正要设计的，也变成：

- 负责什么
- 可以看到什么
- 可以做到哪一步
- 何时主动介入
- 何时保持静默
- 何时必须找人
- 如何审计和撤销

## 聊天框可能只是 Agent 时代的命令行

聊天框足够通用，适合表达模糊目标、临时约束、中途修改和异常。技术早期很有价值。

但它并不适合表达：

- 长任务进度
- 多个并行任务
- 权限范围
- 风险等级
- 审批节点
- 行动历史
- 产物版本
- 可回滚状态

这很像早期计算机的命令行：灵活、能表达几乎一切、适合专家，却把系统复杂性暴露给用户。

当 Runtime 逐渐形成稳定的任务、状态、权限、行动和结果抽象后，新的交互组件才会真正出现：任务卡片、计划视图、审批节点、风险面板、行动时间线、产物空间、权限中心、回滚历史、动态决策界面。

**聊天不会消失，但会从整个产品退回到一种补充控制方式。**

## 主动服务需要一套产品契约

前面把系统拆成 Runtime、业务世界和环境还不够。静默服务还需要人与 Agent 之间的交互契约：

**Service Policy = Trigger + Autonomy + Interruption + Escalation + Accountability**

- **Trigger**：什么事件、状态或长期目标触发服务
- **Autonomy**：哪些动作可自行执行，哪些只能准备方案
- **Interruption**：什么时候值得通知，什么时候应保持静默
- **Escalation**：高风险、信息不足、价值冲突或越权时如何升级给人
- **Accountability**：信息来源、行动、结果和状态如何被保存、解释和撤销

这套 Service Policy，可能成为未来 Agent 产品最核心的设计对象。

## 它高度依赖 Runtime，但不必再写成半本手册

主动服务的风险远高于普通对话。答错一句话，和误改日程、误付费用、误操作车辆，不是同一量级。

因此它至少需要：持续状态、事件感知、风险分级、可验证行动、恢复回滚、Trace 与责任。

Runtime 与交互也会共同演化：产品出现“暂停”“撤销”“为什么这样做”“只在关键节点问我”，就会反过来要求 Checkpoint、副作用声明、决策轨迹和审批策略。

现在聊天框占主导，很可能不是终局已定，而是底层任务对象和控制能力还不够稳。

## 显性 Agent 与隐性 Agent

产品形态变化，也不一定会先变成一个全新消费者界面。

Agent 可能先改变客服后台、项目跟踪、软件开发、数据分析、企业流程和异常处理。用户看到的仍是普通页面、报告或 App，但后台已从人工流程变成 Agent 驱动的动态执行。

- **显性 Agent**：用户明确在委托和监督
- **隐性 Agent**：用户仍用传统产品，后台已由 Agent 组织完成

后者规模可能更大，却不一定长得像“Agent 界面”。

## 这一篇的结论

Agent-native 产品的核心问题，不再是：

> 用户如何与模型对话？

而是：

**用户如何委托责任、设置权限、定义打扰边界、处理异常，并验收真实世界中的结果？**

责任域是产品单位，聊天框是过渡入口，Service Policy 是真正要设计清楚的契约。
