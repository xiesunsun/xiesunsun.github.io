**如果只看最终答案，很多完全不同的故障，最后都会表现成同一种失败。**

所以我越来越相信，Agent 的评价对象不该只是最终结果，而应是完整轨迹。

## 评价完整轨迹，而不是只打一个总分

真正该看的执行链大致是：

目标定义 → 信息获取 → Context 构造 → 模型决策 → 工具执行 → 状态更新 → 结果验证

未来成熟的评价体系，大概不会只剩一个分数，而会同时看：

- 最终任务结果
- 决策和工具调用过程
- 环境最终状态
- 成本与延迟
- 错误恢复
- 人工介入
- 安全与完整性
- 是否存在 reward hacking

这里真正要提升的，不是“指标数量”，而是**可辨识性**：

**当 Agent 的表现发生变化时，我们能不能知道究竟是什么造成的？**

没有可辨识性，所谓优化就很容易变成玄学调参。

## 自进化首先是实验基础设施问题

Hermes 这类系统已经展示了不少有意思的方向：从执行轨迹里生成 Memory，提炼 Skill，优化 Prompt，甚至导出训练数据。这些都很值得看。

但我并不觉得，这就证明“自进化”已经解决了。

因为系统在自动修改自己之前，必须先回答：

- 为什么失败
- 应该改哪一层
- 修改后是否真的更好
- 提升是否只是环境变化带来的假象
- 是否对评测集过拟合
- 是否损害了其他能力

所以可靠自进化的前提其实很朴素：

**Agent 的运行状态必须可观察、可版本化、可回放、可比较、可回滚。**

没有这些能力，系统只能自动“变化”，不能证明自己是在“进化”。

## Agent 需要类似 Git 的完整状态版本

传统 Git 保存代码。但 Agent 的行为取决于更多东西：

- 模型及推理配置
- Prompt 和 Skill
- Memory
- Context 构造逻辑
- Harness 代码与配置
- 工具 Schema 和版本
- 评价器版本
- 环境和任务状态

可以把这理解成一个 `Agent Commit`。

但只有快照还不够。你还得记录完整事件流：用户输入、Context 来源、最终发给模型的消息、模型决策、工具调用与返回、状态变化、验证结果。

所以我现在越来越觉得，可靠的 Agent 基础设施更接近：

**Git + Event Sourcing + Distributed Tracing + Replay Simulator**

## 回放追求的是因果可比较，不是 100% 复制现实

真实环境永远无法被百分之百恢复。模型有随机性，工具和外部数据会变，历史状态也可能缺失。

所以回放的目标，不该是逐 token 得到相同输出，而是：

**尽可能固定主要条件，只改变需要研究的组件，从而判断它是否真正产生了改善。**

至少可以区分几种回放：

1. **模型决策回放**：冻结 Context 和工具结果，只比较模型或 Prompt
2. **Harness 回放**：重新构造 Context，但使用冻结的历史工具返回
3. **环境模拟回放**：在 Mock 或数字孪生环境中重新执行工具
4. **真实环境 Shadow Replay**：候选版本影子运行，不影响实际用户

相应地，还需要“回放保真度”：哪些状态被恢复了，哪些缺失了，这些缺失又可能怎样影响结论。

没有保真度意识的回放，很容易给人一种“好像验证过了”的错觉。

## 两个循环：业务执行，和能力学习

一个成熟 Agent，至少应该同时跑两个循环。

业务执行：任务 → 决策 → 工具执行 → 业务结果

能力学习：运行轨迹 → 失败发现 → 成因归类 → 干预选择 → 评测验证 → 系统升级

真正的数据飞轮，不是简单保存日志，而是把真实轨迹转化成：有明确任务定义、完整状态、失败成因、验证结果和干预标签的数据。

只有经过归因后，剩下的“模型能力残差”，才适合进入 SFT、偏好优化或强化学习。否则，你直接拿 badcase 去训模型，可能只是在让模型补偿错误的工具、Context 或产品设计。

## 这一篇的结论

**先让世界能被保存，让历史能被回放，让修改能被归因，让进化能被验证。**

否则所谓自进化，不过是不受控的自动改动。
