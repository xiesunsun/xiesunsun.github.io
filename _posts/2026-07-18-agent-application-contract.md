**如果通用 Runtime 与业务层要稳定分离，中间必须存在一个清晰契约。**

这个契约不该只是一个 Prompt 文件，也不该只是工具列表。它至少要说清楚五件事：任务、状态、Context、行动副作用、评价。

## 1. 任务契约

一个任务需要定义：

- 初始输入
- 所需状态
- 可采取的行动
- 完成条件
- 失败条件
- 时间和成本预算
- 是否允许人工介入

没有明确任务契约，Agent 只能靠模型自己猜任务是否完成。猜对几次不代表系统可靠。

## 2. 状态契约

业务层需要定义：

- 世界中有哪些状态
- 状态从哪里获得
- 状态何时失效
- 哪些任务依赖哪些状态
- 哪些状态可以写入 Memory
- 哪些状态涉及隐私和权限

Runtime 负责保存和版本化状态；状态的业务含义必须由业务层声明。

否则你会得到一个“什么都能存”的状态仓库，却不知道什么值得信、什么已经过期。

## 3. Context 契约

Context 不应只是任意文本，而应带有：

- 来源
- 时间
- 版本
- 有效期
- 优先级
- 敏感级别
- 注入原因
- 与任务的关系

这样系统才能回答：

**这段信息为什么进入模型？它是否已经过时？Replay 时能否恢复？**

没有这些元信息，Context 工程就会退化成“往窗口里塞字”。

## 4. 行动与副作用契约

一个工具不能只有名称、描述和参数，还应说明：

- 前置条件
- 权限
- 是否只读
- 是否有副作用
- 是否可逆
- 是否幂等
- 风险等级
- 成功后的状态变化
- 如何验证执行结果
- 失败后如何补偿

Agent 操作真实世界时，副作用语义往往比函数调用格式更重要。

工具返回 `success`，不等于真实世界已经变成你想要的样子。

## 5. 评价契约

业务层需要明确：

- 什么是成功
- 什么是不可接受的失败
- 哪些指标必须满足
- 哪些结果需要人工判断
- 哪些风险不能被平均分抵消
- Replay 保真度到什么程度才能下结论

Runtime 可以执行评测，但不能替业务定义真实价值。

## 为什么契约比框架口号更重要

很多人一上来就讨论“统一 Agent 框架”。我觉得更值得先问：

**业务包加载到 Runtime 上时，双方到底承诺了什么？**

- 任务是否可判定完成
- 状态是否可恢复和可审计
- Context 是否可解释和可回放
- 行动是否可验证和可补偿
- 评价是否能支撑版本比较

这些问题答不清，框架再通用，也只是把复杂度藏进配置和 Hook 里。

## 这一篇的结论

**契约是 Runtime 与业务之间的稳定边界。**

它让通用层专注机制，让业务层保留世界语义，也让评测、回放和演进有东西可以抓住。
